项目全局总结

SuperMew 是一个基于 Self-Reflective RAG (CRAG) 理念构建的高级智能问答知识库系统。它突破了传统简单 RAG（单纯相似度召回然后丢给模型回答）的局限，在召回率、上下文完整度、以及大模型防止幻觉上做了大量的工业级工程设计。

FastAPI LangChain LangGraph (状态机编排) Milvus (双路向量数据库) HuggingFace (本地Embedding)

🌟 六大核心工程亮点

  • LangGraph 自治检索纠错：如果初步检索相关性差，系统会自动触发退步提问 (Step-back) 或假设性文档 (HyDE) 生成来扩充查询。
  • 双路混合检索引擎：系统不依赖商业 API，自己在 embedding.py 徒手实现了持久化的 BM25 稀疏向量计算。结合语义密集向量，在 Milvus 中利用 RRF (Reciprocal Rank Fusion) 算法做排名融合，兼顾语义泛化和专有名词精准匹配。
  • 父子层级文档切片 (Parent-Child Chunking)：为了解决检索碎片化问题，系统对文档执行 L1/L2/L3 三级切片。仅把极小的 L3 存入 Milvus，一旦命中，自动向上溯源拉取大段 L2/L1 上下文喂给大模型。
  • 防幻觉系统级 Prompt：严格约束 Agent，要求必须基于知识库结果输出，严禁没结果强答或一回合内陷入死循环查库。
  • 高并发流式穿透体系：RAG 复杂流程非常耗时，系统通过 asyncio.Queue 在 agent.py 中实现了底层引擎状态的实时捕获，在 LLM 思考前就把 “检索进度” 推送给前端。
  • 轻量级无依赖架构：核心算法除了依赖底层库外，基本采用原生手写（如 BM25 增量状态机、原生 SSE 协议交互等），极其透明可控。

🧠 [重点] AI大脑状态机编排 (backend/rag_pipeline.py)

普通的 RAG 流程是直线的：“搜索 -> 组装Prompt -> 大模型输出”。但真实的文档搜索往往会失败（比如用户用了同义词、指代不明）。本系统使用 LangGraph 建立了一个带有自我纠错机制的循环状态图（Self-Reflective RAG）。

核心机制 1：文档裁判官 (Grader)

在初次检索 retrieve_initial 之后，系统不急于让主模型回答，而是先让一个小模型（Grader）去“打分”。如果文档跟问题牛头不对马嘴，直接判定为 no，避免主模型产生幻觉。

核心机制 2：查询变换 (Query Transformation)

被 Grader 驳回后，系统进入 rewrite_question_node。这不是简单的换词，而是采用了高阶策略：

  • HyDE (Hypothetical Document Embeddings)：当问题很概念化时，让模型直接“盲答”写一篇假设性文章，拿这篇文章的向量去搜索。因为答案的行文风格往往和原始文档更接近，相似度极高。
  • Step-back (退步提问)：如果问题是具体的“报错日志”，退回一步提问“这个组件的报错排查原理是什么”，从而召回整体结构文档。

核心代码剖析：LangGraph 有向图编译

这段代码定义了整个 AI 思考的生命周期：

def build_rag_graph():
    # 1. RAGState 是图里流通的共享内存，包含了问题、文档和轨迹
    graph = StateGraph(RAGState)
    
    # 2. 注册图节点（定义执行函数）
    graph.add_node("retrieve_initial", retrieve_initial)
    graph.add_node("grade_documents", grade_documents_node)
    graph.add_node("rewrite_question", rewrite_question_node)
    graph.add_node("retrieve_expanded", retrieve_expanded)

    # 3. 画出固定的单行道
    graph.set_entry_point("retrieve_initial")
    graph.add_edge("retrieve_initial", "grade_documents")
    
    # 4. 【灵魂】条件分叉：根据 grader 评分决定走向
    graph.add_conditional_edges(
        "grade_documents",
        lambda state: state.get("route"), 
        {
            "generate_answer": END, # 裁判给过了：结束图计算，主模型准备输出
            "rewrite_question": "rewrite_question", # 没过：去重写问题
        },
    )
    
    # 5. 重写完了去二次搜索，完了再结束
    graph.add_edge("rewrite_question", "retrieve_expanded")
    graph.add_edge("retrieve_expanded", END)
    
    return graph.compile()

🔢 [重点] 本地增量 BM25 稀疏向量 (backend/embedding.py)

商业向量模型（如 OpenAI、BAAI）生成的叫做“稠密向量”(Dense)，擅长“语义模糊匹配”。但如果在手册里搜 "XYZ-9988C 型号"，稠密向量往往全军覆没。系统在此处硬核地在纯 Python 环境下手写了 BM25 (Sparse) 稀疏向量算法。

硬核点 1：不依赖外接服务的纯净状态机

系统没有外接 ElasticSearch，而是维护了一个 data/bm25_state.json。每次上传/删除新文档，它都会被安全地锁（threading.Lock()）住并重新计算每个词的 doc_freq（逆文档频率）和全库文档总数 total_docs，实现了轻量级的 增量 IDF。

核心代码剖析：TF-IDF 稀疏向量构建

由于 Milvus SPARSE_INVERTED_INDEX 的要求，我们需要将文本转为一个形如 {词汇ID: 得分} 的字典（其中只包含非零分数，这就叫稀疏）。

def _sparse_vector_for_text_unlocked(self, text: str):
    # 1. 拆词并统计当前句子的词频 (Term Frequency, TF)
    tokens = self.tokenize(text)
    doc_len = len(tokens)
    tf = Counter(tokens)
    
    sparse_vector = {}
    for token, freq in tf.items():
        # 词典索引，用于给 Milvus 做倒排 ID
        idx = self._vocab[token]
        
        # 2. IDF 计算（逆文档频率）：如果一个词在全库太常见，得分被暴扣
        df = self._doc_freq.get(token, 0)
        if df == 0:
            idf = math.log((n + 1) / 1)
        else:
            idf = math.log((n - df + 0.5) / (df + 0.5) + 1)

        # 3. 经典 BM25 算法公式
        # numerator: TF 上限抑制 (饱和度参数 k1)
        # denominator: 引入文档长度惩罚 (参数 b)
        numerator = freq * (self.k1 + 1)
        denominator = freq + self.k1 * (1 - self.b + self.b * doc_len / avg)
        
        score = idf * numerator / denominator
        if score > 0:
            # { 维度ID: 分数 } 这就是稀疏向量的本质
            sparse_vector[idx] = float(score)

    return sparse_vector, vocab_changed

核心代码剖析：RRF (Reciprocal Rank Fusion) 算法

这是混合检索最难的一步：Dense 的距离分数（例如 0.81）和 BM25 的得分（例如 25.4）如同苹果和香蕉，没法加权平均。系统巧妙使用了 RRF 算法，彻底抛弃绝对分数，仅使用两边的排名次序来合并。

def hybrid_retrieve(self, dense_embedding, sparse_embedding, top_k=5):
    # 1. 构造语义查询：让它多跑一点候选量 (top_k * 2) 参与后面大乱斗
    dense_search = AnnSearchRequest(
        data=[dense_embedding], anns_field="dense_embedding", limit=top_k * 2
    )
    
    # 2. 构造关键词纯词频查询
    sparse_search = AnnSearchRequest(
        data=[sparse_embedding], anns_field="sparse_embedding", limit=top_k * 2
    )
    
    # 3. 部署 RRF 重排器 (重排分 = 1 / (k参数 + 单路排名))
    # 排名越靠前（单路排名越小），分数越大；如果两路都排名高，分数加成极高
    reranker = RRFRanker(k=60)
    
    # 4. 把并发的两路搜索抛给 Milvus 底层执行，直接获取排好序的金子
    results = self.client.hybrid_search(
        collection_name=self.collection_name,
        reqs=[dense_search, sparse_search], # 多路并发
        ranker=reranker,                    # 使用 RRF 统分
        limit=top_k
    )
    return results

📄 [重点] Parent-Child 文档层级切片 (backend/document_loader.py)

系统用“三级切块”解决了 RAG 永远的痛：切小了（命中率高但模型看不懂上下文），切大了（一整页去匹配，检索全是干扰项）。

切块策略设计与存储分离

文档通过 RecursiveCharacterTextSplitter 滑动窗口变成了三层。但最牛的地方在于：入库 Milvus 的永远只有最小的碎块（L3），大块（L1/L2）只作为本地 JSON 存储着，只带有血缘 ID（parent_chunk_id）。等极精准的 L3 命中后，系统凭 ID 把沉睡的 L1 大段捞起供模型理解。

核心代码剖析：血缘级联切分

class DocumentLoader:
    def _split_page_to_three_levels(self, text, base_doc):
        # 【Level 1 - 宏观层】 约 1200 字符的大章大段
        level_1_docs = self._splitter_level_1.create_documents([text])
        for l1_doc in level_1_docs:
            l1_id = self._build_chunk_id(1) # 给 L1 颁发唯一身份证
            
            # 【Level 2 - 中观层】 基于刚才生成的 L1 内容，切出 600 字符中段
            level_2_docs = self._splitter_level_2.create_documents([l1_doc.page_content])
            for l2_doc in level_2_docs:
                l2_id = self._build_chunk_id(2) # 给 L2 颁发身份证
                
                # 【Level 3 - 微观层】 基于 L2 内容，切出 300 字符句子
                level_3_docs = self._splitter_level_3.create_documents([l2_doc.page_content])
                for l3_doc in level_3_docs:
                    # 最终装箱打包，这个字典里的数据最终流入库里
                    root_chunks.append({
                        "text": l3_doc.page_content,
                        "chunk_id": self._build_chunk_id(3), # 自己的 ID
                        "parent_chunk_id": l2_id,  # 【外键】：我老爸是谁
                        "root_chunk_id": l1_id,    # 【外键】：我爷爷是谁
                        "chunk_level": 3,          # 我是第 3 层
                    })

设计亮点：异步队列解决并发通信难题

在标准的 Python 流程中，Agent 执行 search_knowledge_base 时整个线程会阻塞。为了让前端看到 "🔍正在重写查询..." 这种底层工具发出的进度，我们利用了 asyncio.Queue：

async def chat_with_agent_stream(user_text, user_id, session_id):
    # 统一输出通道：模型Token和底层工具状态都往这里丢
    output_queue = asyncio.Queue()

    # 这个代理类会塞进全局，供深层的 LangGraph 执行时实时把进度塞进队列
    class _RagStepProxy:
        def put_nowait(self, step):
            output_queue.put_nowait({"type": "rag_step", "step": step})

    # 开启后台独立协程执行大脑推理
    async def _agent_worker():
        async for msg in agent.astream(...):
            await output_queue.put({"type": "content", "content": msg})
        await output_queue.put(None) # 完成信号

    asyncio.create_task(_agent_worker())

    # 主循环流式返回给前端
    while True:
        event = await output_queue.get()
        if event is None: break
        yield f"data: {json.dumps(event)}\n\n"

🚪 工具网关与状态逃逸 (backend/tools.py)

这个模块扮演了 LangChain Agent 和 LangGraph RAG 状态机 之间的桥梁。

状态逃逸设计

普通的 LangChain Tool 只能向大模型返回一段 String。但复杂的 RAG 过程产生了大量珍贵的中间日志（rag_trace）。工具在这里提取出干干净净的文本给模型，而把 rag_trace 偷偷扔进了全局对象，等待上面提到的主循环抽走传给前端。

📝 双重向量并行压入 (backend/milvus_writer.py)

在数据真正落盘到 Milvus 之前调度 Embedding 模型。“一文两算”模式的核心发起地。

📦 本地块存储 (backend/parent_chunk_store.py)

实现本地化的轻量级 JSON 存储库 ParentChunkStore，主要负责在 RAG 返回 L3 碎片数据后，根据 parent_chunk_id 高速取回完整长文本。

🖥️ 前端纯净渲染 (frontend/index.html)

采用原生 HTML/JS 开发，包含对 SSE 协议 (Server-Sent Events) 的流式解析渲染。

🌐 API 核心路由 (backend/api.py)
